# kahyad/internal/policy ‚Äî W3-02 wire schema

`POST /policy/check` `{tool_name, tool_input, scope?, task_id, trace_id}` ‚Üí `{decision: "allow"|"needs_approval"|"deny", reason?, rule, pending_approval_id?, token?}`. `class` is never client-supplied ‚Äî resolved from the loaded `policy.yaml`. `token` appears only on `allow` for a non-R (side-effectful) class; `pending_approval_id` only on `needs_approval`.

`pending_approval_id` (post-security-review amendment) is a server-issued, single-use, DB-tracked reference: `Engine.Check` inserts a `pending_approvals` row (32 random bytes hex id, bound to the RESOLVED tool/class/scope/task_id/trace_id/approved_bytes_hash, 10-minute TTL) and returns its id ‚Äî it is opaque and carries no caller-decodable meaning. `Engine.Approve`/`Deny` look the row up by id and atomically single-use-consume it (`consumed_at IS NULL`); a forged, expired, or already-consumed id is rejected outright, before any token is minted or any bookkeeping runs. W3-06 additionally persists the row's exact `tool_input` bytes (not just their hash) so `kahya approvals`/`kahya approve <id>` can render the real byte-exact WYSIWYE diff ‚Äî see `Engine.ListPendingApprovals`/`GetPendingApprovalDetail`, both read-only (never consume).

`POST /policy/consume-token` `{token, tool, class, scope, task_id, trace_id, tool_input}` ‚Üí `{ok, error?}`. Side-effectful tools MUST call this, with the EXACT bytes they are about to execute, immediately before executing. Single-use; any failure demotes the token's REAL bound `(tool,class,scope)` ‚Äî persisted on `approval_tokens` at mint time and recovered by `token_hash` ‚Äî never the `tool`/`class`/`scope` this request body happens to claim, and ledgers `token_verify_failed`. A hash matching no token row at all demotes nothing (there is no real triple to punish) but still ledgers. W3-06: the hash compared is `approvedBytesHash`, which routes `tool_input` through `kahyad/internal/canon.CanonicalizeBytes` (NFC-normalize, strip bidi/zero-width/other-Cf control code points ‚Äî confusables are flagged, never rewritten) before hashing ‚Äî an NFD-vs-NFC difference in the same path collapses to an identical hash, while a genuine byte mutation (a trailing space, a homoglyph swap) still changes it. `mintToken`'s own token-hash (the lookup key derived from the random token bytes themselves) is a separate, plain `rawBytesHash` ‚Äî never canonicalized, since a random credential carries no WYSIWYE meaning.

`POST /policy/feedback` `{kind: "approve"|"deny"|"undo", pending_approval_id?, surface?, trace_id?}` ‚Üí `{ok, token?, error?}`. `approve` mints a token (W3 requires `surface:"local"`, checked against the pending_approvals row's real class ‚Äî a rejected non-local W3 approval attempt does NOT consume the id, so it stays usable for a later local approval, and ledgers `w3_nonlocal_approval_rejected`, Go-side, so Telegram/W3-07 can never be wired to approve a W3 action); `deny`/`undo` demote.

`GET /policy/approvals` ‚Üí `{approvals: [{id, tool, class, scope, summary, minted_at}]}` (W3-06, `kahya approvals`) ‚Äî every not-yet-consumed, not-yet-expired pending approval, oldest first. `GET /policy/approvals?id=<id>` ‚Üí the single row's full rendered WYSIWYE diff (`kahya approve <id>`'s own lookup ‚Äî read-only, does not consume).

`GET /policy/state` ‚Üí `{states: [{tool, class, scope, level, consecutive_approvals, updated_at}]}`.

`POST /policy/promote` `{tool, class, scope}` ‚Üí `{level}` ‚Äî the only path that ever raises a level (`kahya autonomy promote`).

`POST /policy/undo` `{trace_id}` ‚Üí `{ok, tool, task_id}` ‚Äî triggers an open undo window; recipe execution is the owning tool's job (W3-03).
